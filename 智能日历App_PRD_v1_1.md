## 一、产品定位

**一句话描述：** 一个帮助自律困难型用户自动把目标排进空闲时间、并像课程表一样主动提醒执行的私人日程助手。

**核心价值：** 用户只需输入"我想做什么、需要多久、截止什么时候"，App 自动完成排程，不需要用户手动想"这件事我什么时候做"。

**目标用户：** 当前仅设计给开发者本人使用（单用户 App），暂不考虑多用户/分享场景。

## 二、核心概念定义

### 2.1 日类型

| 类型 | 定义 | 可用时间 |
|------|------|----------|
| 工作日 | 有实习安排的日子 | 当前阶段配置的下班后时段 |
| 休息日 | 无实习安排的日子 | 全天（用户配置的起止时间） |

**日类型规则：**
- 默认跟随中国法定节假日日历
- 用户可对任意单天手动覆盖（如"这个周三我请假，改为休息日"）

### 2.2 时间阶段（Phase）

用户生活状态会阶段性变化，App 支持配置多个时间阶段，每个阶段有独立的可用时间配置：

```
阶段示例：
- 实习期（至 2025-09-01）：工作日可用时间 19:30–22:30，休息日 09:00–22:00
- 实习结束后：工作日可用时间 08:00–22:00，休息日 08:00–22:00
```

切换阶段时，排程引擎自动按新规则重新计算未来的安排。

### 2.3 任务类型

**类型 A：项目任务（有 deadline）**

用户需要在某个截止日期前完成的任务，支持以树状结构不断细化拆分——顶层任务可拆出子任务，子任务还可以继续拆分，每一层级都可以独立设置 deadline、时长和排程，并在后续随时增删调整。排程引擎以叶子节点（最末级、不再拆分的子任务）为实际执行单元来分配时间块；父任务的 deadline 作为其所有子任务的默认上限约束，子任务可单独设置更早的 deadline。

字段（每个节点通用）：
- 任务名称
- 总时长（小时或分钟，叶子节点必填；表示这个任务需要多长时间完成）
- 每个时间块时长（分钟，叶子节点必填；表示一次集中工作的时长，如 180分钟）
- Deadline（截止日期；子任务 deadline 不得晚于父任务）
- 分布偏好：集中（同一任务连续排）
- parentId（为空时表示顶层任务）

**计算方式：** App 自动计算总块数 = ⌈总时长 ÷ 每块时长⌉，例如 8 小时任务，每块 3 小时 → 需要排 3 个时间块（实际占用 9 小时）

**类型 B：循环习惯（有周期）**

用户希望在一段时间内保持的周期性行为。

字段：
- 习惯名称
- 每次时长（分钟）
- 每周频次（次/周）
- 循环周期（如"接下来 4 周"）
- 到期后行为：弹出询问是否续期 + 可修改频次

## 三、核心功能需求

### 3.1 初始化设置流程

首次打开 App 时，引导用户完成：

1. 设置当前时间阶段名称及可用时间（工作日 / 休息日各一段）
2. 确认默认节假日日历（中国法定）
3. 引导添加第一个任务（可跳过）

### 3.2 任务管理

**添加任务：**

- 用户手动填写结构化表单（任务名、总时长、每块时长、deadline 或频次/周期）
- 填写完成后，App 计算排程预览（自动计算需要的时间块数），展示"拟安排到以下时间段"列表
- 用户审阅后点击「应用」才正式写入日程；可点击「重新排」或手动调整个别块

**修改任务：**

- 可修改任务的任意字段
- 修改后触发受影响部分的重新排程（仅影响未完成的块）

**删除任务：**

- 删除任务时，其所有未完成的时间块一并清除
- 已完成的历史记录保留

### 3.3 排程引擎规则

**时间块分配逻辑：**

1. 在当天可用时间段内，按块时长切分可用空档
2. 项目任务：集中排，同一任务的块尽量排在相邻的时间段（同天优先，其次相邻天）
3. 循环习惯：均匀分布在周期内，避免同一天排两次同一习惯
4. 项目任务与循环习惯发生冲突时（空档不足）：**不自动决策，提醒用户手动选择优先级**

**排程视野：**
- 项目任务：从今天起到 deadline 前，找足够的空闲块
- 循环习惯：在用户设定的循环周期内排满

**顺延规则：**
- 某时间块被标记为未完成（手动标记或用户跳过），自动顺延到下一个同类型空闲时段
- 顺延前展示"已顺延到 XX 时间"的通知

### 3.4 日程视图

**主视图（今日视图）：**

- 默认打开页面
- 展示今天所有已排的时间块，按时间顺序排列
- 每个时间块显示：任务名、开始/结束时间、所属任务类型标识
- 时间块状态：待执行 / 进行中 / 已完成 / 已顺延

**周视图：**

- 可从今日视图切换进入
- 展示本周每天的时间块概览

**操作：**

- 手动勾选时间块为"已完成"
- 长按时间块可手动拖动调整时间（在同天可用时段内）

### 3.5 通知系统

| 通知类型 | 触发时机 | 内容 |
|----------|----------|------|
| 任务提醒 | 时间块开始前 5 分钟 | "5 分钟后开始：[任务名]（[时长]分钟）" |
| 顺延通知 | 用户标记未完成后 | "[任务名] 已顺延至 [新时间]" |
| 循环任务到期 | 循环周期结束当天 | "[习惯名] 本轮已结束，是否继续下一轮？" |
| 冲突提醒 | 排程时发现空档不足 | "新任务与 [已有任务] 存在时间冲突，请手动调整优先级" |

**通知要求：**
- 使用 Android 精确闹钟权限（`USE_EXACT_ALARM`）
- 首次安装时引导用户将 App 加入电池白名单，确保通知不被系统杀掉

### 3.6 桌面小组件（Android Widget）

显示"当前 / 即将进行"的任务状态，无需打开 App 即可知晓。

**小组件内容：**
- 当前时间块：任务名 + 剩余时间（如"普拉提 · 还剩 23 分钟"）
- 若当前无任务：显示下一个任务名 + 距开始时间（如"小论文第4章 · 19:30 开始"）
- 若今日无任务：显示"今天没有安排 🎉"

**小组件规格：**
- 尺寸：4×1（标准横向小组件）
- 点击小组件跳转 App 今日视图
- 刷新频率：每分钟更新一次剩余时间

## 四、数据架构（草案）

### 4.1 核心数据模型

```
Phase（时间阶段）
  - id
  - name（如"实习期"）
  - startDate / endDate
  - workdayFreeStart / workdayFreeEnd（HH:mm）
  - holidayFreeStart / holidayFreeEnd（HH:mm）

DayOverride（单日覆盖）
  - date
  - type（workday | holiday）

Task（任务节点，类型A，支持树状结构）
  - id
  - parentId（null 表示顶层任务）
  - name
  - totalDuration（分钟，叶子节点必填）
  - blockDuration（分钟，叶子节点必填）
  - deadline
  - completedBlocks（已完成的时间块数）
  - status（active | completed | cancelled）
  // App 自动计算：totalBlocks = ceil(totalDuration / blockDuration)
  // 排程引擎只对叶子节点（无子任务的节点）生成 TimeBlock
  // 父节点 deadline 作为子任务的上限约束

Habit（循环习惯，类型B）
  - id
  - name
  - durationPerSession（分钟）
  - timesPerWeek
  - cycleStart / cycleEnd
  - status（active | expired | cancelled）

TimeBlock（已排的时间块）
  - id
  - sourceId（Task.id 或 Habit.id）
  - sourceType（task | habit）
  - date
  - startTime / endTime
  - status（pending | done | postponed）
```

### 4.2 数据存储

- **本地：** SQLite（通过 Expo SQLite 或 WatermelonDB）
- **云同步：** Firebase Firestore（免费档，单用户数据量极小）
- **同步策略：** 本地优先，有网时自动同步；换手机后登录账号恢复全部数据

## 五、MVP 范围（第一版）

### ✅ 包含

- 时间阶段配置
- 法定节假日 + 单日覆盖
- 项目任务（类型A）的添加、排程预览、应用、勾选完成
- 循环习惯（类型B）的添加、排程、到期提醒
- 今日视图
- 任务开始前 5 分钟通知
- 未完成自动顺延
- 冲突时提醒用户手动处理
- 本地存储 + Firebase 云同步
- 桌面小组件（4×1）

### ❌ 不包含（留待后续版本）

- 自然语言输入（"帮我排普拉提一周两次"→AI 自动解析）
- 周视图（MVP 后加）
- 任务统计 / 完成率图表
- 多用户 / 分享功能
- iOS 版本

## 六、技术选型
